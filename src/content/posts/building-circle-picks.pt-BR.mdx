Recentemente lancei o [Circle Picks](https://www.circlepicks.app), uma plataforma social onde você compartilha e descobre lugares através de pessoas em quem confia. Pense nele como o guia curado dos seus amigos para restaurantes, cafés, bares e hotéis. Sem reviews falsas, sem conteúdo patrocinado. Apenas indicações reais de pessoas reais.

O app está no ar na web, e a versão mobile está em beta. Ele tem autenticação, busca de lugares com GPS, upload de fotos, sistema de seguidores, salvos, feed de exploração, i18n em dois idiomas e painel de moderação para admin. E eu construí praticamente sozinho, com IA como meu co-piloto.

Veja como eu fiz, dividido em quatro fases.

## 1. Idealização, Planejamento e Organização

Antes de escrever uma linha de código, investi tempo estruturando minhas ideias de uma forma que tanto eu quanto uma IA pudéssemos entender.

Criei um conjunto de arquivos de contexto: documentos JSON e Markdown que descrevem cada aspecto do projeto. A estratégia do produto, o stack técnico, as specs de UX, o schema do banco, as rotas da API, os tokens de design. Tudo.

Na prática, esses arquivos são assim:

- **front.strategy.json** - Objetivos do produto, casos de uso, escopo do MVP (o que entra, o que fica de fora), critérios de sucesso
- **front.tech.json** - Stack completo com versões, todos os modelos TypeScript, tabelas do banco, inventário de APIs, princípios de arquitetura
- **front.ux.json** - Modelo de navegação, specs de cada tela, tokens de design (cores, tipografia, espaçamento), regras de interação
- **front.setup-plan.json** - 18 fases de implementação, SQL de migração, princípios críticos
- **index.json** - Um hub de navegação mestre com breadcrumbs e referências cruzadas entre todos os docs

Esses arquivos se tornaram a fonte única de verdade do projeto inteiro. Quando eu começava uma nova feature, não precisava re-explicar o projeto do zero. A IA tinha o contexto completo: o que o app faz, como está estruturado, por que certas decisões foram tomadas.

O insight principal foi esse: **o tempo que você investe organizando suas ideias no começo se paga exponencialmente quando você está construindo com IA.** Um arquivo de contexto bem estruturado vale mais que cem prompts.

Eu também mantive um arquivo `CLAUDE.md` na raiz do projeto, um arquivo de instruções vivo que diz ao Claude Code como trabalhar com esse projeto específico. Inclui o stack, estrutura do projeto, convenções de nomenclatura, padrões de rotas, regras de estilo e convenções de commit. Toda vez que o Claude abre o projeto, ele lê esse arquivo primeiro e sabe exatamente como se comportar.

## 2. Implementação

Com o planejamento feito, a implementação andou rápido. Muito rápido.

Usei o Claude Code como minha ferramenta principal de desenvolvimento. Não como um autocomplete chique, mas como um parceiro de desenvolvimento de verdade. Eu descrevia o que queria, apontava para os arquivos de contexto certos, e ele escrevia código que seguia os padrões do projeto, usava os componentes corretos, respeitava as convenções de rotas e até lidava com i18n em português e inglês.

O segredo eram os arquivos de instrução. Cada projeto tem seu próprio `CLAUDE.md` com regras específicas:

- Sempre usar `Link` de `@/i18n/routing`, não de `next/link`
- Usar `cn()` para merge condicional de classes
- Seguir o sistema de rotas de três camadas (`ROUTE_PATHS`, `routes`, `routePaths`)
- Usar o client Supabase correto para cada contexto (server vs browser)
- Commits convencionais, minúsculos, mensagens curtas

Isso significava que eu raramente precisava corrigir o output da IA. Simplesmente funcionava dentro das convenções do projeto.

Vale mencionar: meu amigo Hel Rabelo construiu uma ferramenta chamada [TokenCentric](https://www.tokencentric.app/), um app feito para organizar suas instruções de IA por projeto e gerenciar seus arquivos `CLAUDE.md` em múltiplos codebases. Tem sido muito útil para manter tudo organizado, especialmente quando você está lidando com vários projetos com stacks e convenções diferentes.

### As decisões técnicas que tornaram isso possível

Escolhi um stack que mantivesse as coisas simples e rápidas para um MVP:

- **Next.js 16 com App Router** como framework full-stack
- **Supabase** para autenticação, banco PostgreSQL e storage de arquivos, sem precisar configurar nenhuma infraestrutura backend separada
- **API Routes do Next.js** como REST API. Essa foi grande. Em vez de manter um backend separado, usei as rotas de API dentro do próprio Next.js. Para um MVP, é perfeito. Um codebase, um deploy, nenhuma infra extra pra gerenciar
- **React Query** para gerenciamento de estado do servidor
- **Tailwind CSS** com design tokens customizados para estilo consistente
- **next-intl** para internacionalização. O app roda em português (padrão) e inglês

Também fiz escolhas práticas que economizaram tempo: nenhuma biblioteca de ícones (tudo SVG inline), nenhuma lib pesada de formulários exceto onde necessário, e compressão de imagens no client antes do upload.

## 3. Testes, Validação e Refinamento de UX

Construir features é uma coisa. Fazer elas ficarem realmente boas é outra.

Essa fase foi sobre testar cada fluxo de ponta a ponta, refinar a UX e garantir que o banco estivesse sólido.

**Deduplicação de lugares** foi um dos problemas mais complicados. Quando vários usuários recomendam o mesmo restaurante, você não quer 5 entradas duplicadas. Construí um sistema de matching em múltiplos níveis: match exato por Google Place ID, match case-insensitive por nome + cidade + país, proximidade de coordenadas (~500m) com similaridade de nome, e até normalização de nomes de países (Brasil → Brazil). Quando você adiciona um lugar que já existe, o app mostra um modal com a contagem de recomendações e tags mais usadas, deixando confirmar que está adicionando ao mesmo lugar.

O **fluxo de Quick Add** foi desenhado com um objetivo: salvar um lugar em ~15 segundos. GPS detecta sua localização, você busca o lugar, escolhe faixa de preço, seleciona tags, e pronto. Campos opcionais como descrição e fotos estão lá se quiser, mas nunca bloqueiam o fluxo principal.

**Controles de visibilidade** adicionaram uma camada social que torna o app mais voltado para comunidade. Cada experiência pode ser pública ou só para amigos. O feed respeita isso: você vê suas próprias independente, as públicas da comunidade, e as de amigos de quem você segue. Isso é enforçado no nível do banco com Row Level Security, não só na interface.

A feature **"Outros que indicaram"** acabou sendo uma das melhores adições. Quando você vê uma experiência, aparece quem mais recomendou o mesmo lugar. Cria aquela sensação de validação ("ah, 3 dos meus amigos também amam esse lugar") e incentiva a descoberta por gosto compartilhado.

Também construí um **painel de moderação admin** para qualidade de conteúdo. Experiências podem ser desativadas com um motivo, e o usuário é notificado. Essencial para qualquer plataforma comunitária, mesmo no estágio de MVP.

## 4. Desenvolvimento Mobile

Essa foi a fase que mais me deixou apreensivo. React Native não é meu stack principal. Sou engenheiro frontend focado em web.

Mas tinha duas coisas a meu favor: uma vantagem inicial de um amigo, e o mesmo workflow com IA que tornou o desenvolvimento web rápido.

Pedro, um amigo e desenvolvedor React Native, criou o [PHL React Native Boilerplate](https://boilerplate.pedrohlima.com/) que me deu uma vantagem enorme. Ele já vem com Expo SDK 54, navegação com Expo Router, Zustand + MMKV para estado persistente, NativeWind (Tailwind para React Native) com dark mode, React Query, formulários com React Hook Form + Zod, e i18n pronto. Também segue um padrão de clean architecture com camadas separadas de apresentação, domínio, dados e core. Em vez de lutar com o setup inicial do React Native (que pode ser doloroso se você não está acostumado), rodei `npx create-expo-app -t phl-rn-boilerplate` e tinha uma base pronta para produção em minutos.

A partir daí, apliquei a mesma abordagem que usei na web:

- Criei arquivos de contexto específicos para o projeto mobile
- Configurei um `CLAUDE.md` com convenções React Native
- Usei Claude Code para construir features, aproveitando minha experiência crescente com React Native e o contexto do projeto web

O app mobile precisava ter exatamente as mesmas features que o web. Mesma auth, mesmo feed, mesmo fluxo de adicionar, mesmo explorar, mesmos salvos, mesmo perfil. O fato de eu estar usando API Routes do Next.js como backend tornou isso direto. O app mobile simplesmente chama os mesmos endpoints REST. Nenhum backend separado para construir ou manter.

Supabase resolveu o trabalho pesado em ambas as plataformas:

- **Autenticação:** Email/senha com gerenciamento de sessão
- **Banco:** PostgreSQL com políticas RLS garantindo segurança independente de qual client faz a requisição
- **Storage:** Upload de imagens para fotos de experiências e avatares

Algumas coisas que me deixaram particularmente satisfeito no mobile:

- Localização GPS funciona nativamente no Quick Add, com a mesma cascata (GPS → fallback por IP → seleção manual)
- Compressão de imagens antes do upload, mantendo o app rápido e custos de storage baixos
- Controles de visibilidade (público vs só amigos) consistentes entre plataformas
- A camada de API compartilhada eliminou problemas de paridade. Os dois apps falam com o mesmo backend

## O que aprendi

Construir o Circle Picks me ensinou algumas coisas:

1. **IA funciona melhor quando você dá estrutura.** Arquivos de contexto, arquivos de instrução e convenções claras transformam a IA de um autocomplete de código em um parceiro de desenvolvimento real.
2. **Você não precisa de um backend separado para um MVP.** API Routes do Next.js + Supabase me deram uma solução full-stack completa com um único deploy.
3. **Comece pelo chato.** Autenticação, schema do banco, convenções de rotas. Acertar isso cedo significou poder ir rápido depois sem refatorar.
4. **A comunidade em torno de ferramentas de IA é real.** Ferramentas como [TokenCentric](https://www.tokencentric.app/) do Hel Rabelo, o [PHL Boilerplate](https://boilerplate.pedrohlima.com/) do Pedro. O ecossistema de devs construindo ferramentas para melhorar desenvolvimento assistido por IA está crescendo rápido.
5. **React Native é acessível se você tem o suporte certo.** Um bom boilerplate, arquivos de contexto sólidos e uma IA que entende seu projeto compensam muita falta de experiência em um stack específico.

O [Circle Picks](https://www.circlepicks.app) está no ar na web, e o app mobile está chegando em breve. Se você quer saber como seus amigos descobrem ótimos lugares, experimenta.

E se você está construindo algo com IA, invista nos seus arquivos de contexto. É o multiplicador de produtividade mais subestimado que eu encontrei.
